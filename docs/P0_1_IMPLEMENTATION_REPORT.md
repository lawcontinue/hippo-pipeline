## ❌ 关键问题

### 问题 1: 批量处理性能极差 ⚠️⚠️⚠️

**测试数据**:
- Batch Size: 2
- Step 1 时间: **64.454 秒**
- **tok/s: 0.02**（几乎为 0）
- 对比基线: 8.29 tok/s

**性能差距**: **-99.8%**（严重退化）

### 问题 2: 根因分析

**可能原因**:
1. **逐个 token forward**（未真正并行）
   - 当前实现：循环处理每个 token
   - 问题：无法利用 GPU 并行
   - 代码：`for i in range(batch_size): hi = forward(...)`

2. **KV cache 无法批量**
   - 当前实现：每个 token 的 offset 不同
   - 问题：无法共享 KV cache
   - 结果：逐个处理，性能退化

3. **网络延迟累积**
   - 当前实现：等待 2 个 requests 后才处理
   - 问题：第一个 token 等待第二个 token
   - 结果：延迟翻倍

## 💡 关键洞察

### 洞察 1: 批量处理需要真正的 GPU 并行 ⭐⭐⭐⭐⭐

**错误假设**:
> "累积 2-8 个 tokens 后批量处理，GPU 会自动并行"

**现实**:
> MLX 的 forward_layer 需要逐个 token 处理（因为 KV cache 依赖）

**解决方案**:
- 需要重新设计 forward_layer，支持真正的批量输入
- 或者：接受当前限制，寻找其他优化方向

### 洞察 2: R1 forward 是瓶颈，但无法简单优化 ⭐⭐⭐⭐

**瓶颈分析**:
- R1 forward（23 层）: 161ms（主要瓶颈）
- 但无法通过批量处理优化（KV cache 限制）

**替代方案**:
- 减少层数（如 30-47，vs 25-47）
- 使用 fused op（如 MLX 的优化 kernel）
- 优化每一层的实现

### 洞察 3: 架构限制可能需要架构变更 ⭐⭐⭐⭐⭐

**核心问题**:
> Hippo 的分片架构（R0: 0-24, R1: 25-47）可能不是最优的

**替代架构**:
- Pipeline Parallelism（每一层分片，而非 layer 分组）
- Tensor Parallelism（每一层内部分片）
- 或者：接受当前性能，寻找其他优化点
