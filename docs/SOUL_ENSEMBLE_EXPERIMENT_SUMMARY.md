## 🎯 核心发现

### 1. 共识 = 高置信度 ✅

**6 次共识全部正确**（英文 2 次 + 中文 4 次）

| 题目 | 版本 | 共识答案 | 正确? |
|------|------|----------|-------|
| 5台机器做100零件 | 英文 | 5分钟 | ✅ |
| SaaS年收入 | 英文 | ~50 | ✅（ARPU部分） |
| 蝙蝠球 | 英文 | 0.05 | ✅ |
| SaaS年收入 | 中文 | 5 | ⚠️（提取失败） |
| 瓶装空气 | 中文 | 100 | ✅ |
| 蝙蝠球 | 中文 | 0.05 | ✅ |
| 象棋+篮球 | 中文 | 16 | ✅（开放式） |

**结论**: Stage 1 共识时可以直接输出，高置信度。

### 2. 分歧 → 仲裁修正（大部分情况）

**英文版 5 次分歧**：
- ✅ 3 次忒弥斯仲裁修正为正确（17只羊、蝙蝠球、3加仑5加仑）
- ❌ 2 次开放式问题（瓶装空气、象棋+篮球）

**中文版 4 次分歧**：
- ✅ 1 次忒弥斯仲裁修正为正确（100台机器）
- ❌ 1 次仲裁错误（17只羊——"All but 9 die" 翻译歧义）
- ⚠️ 2 次答案提取失败（3加仑5加仑、电商ROAS）

### 3. Code vs Crit 能力对比

| SOUL | 推理正确率 | 特点 |
|------|-----------|------|
| **Code 💻** | 90%+ | 零废话，步骤化，数学题准确 |
| **Crit ⚖️** | 60-70% | 假设验证能力强，但计算易错 |

**典型案例**：
- **蝙蝠球题**：
  - 英文版：Code 对（$0.05），Crit 错（$1.10）→ 仲裁修正
  - 中文版：Code + Crit 都对（0.05元）→ 共识 ✅
- **17只羊题**：
  - 英文版：Code 对（9），Crit 错（8）→ 仲裁修正
  - 中文版：Code 错（8），Crit 指出歧义 → 仲裁错误（理解为歧义而非9）

### 4. 中英文对比

| 指标 | 英文版 | 中文版 |
|------|--------|--------|
| 共识率 | 25% | **50%** ⭐ |
| 可验证题正确率 | 100% (5/5) | 67% (2/3) |
| 蝙蝠球题 | Code 对，Crit 错 | **Code + Crit 都对** ✅ |
| 平均Token | 464-513 | 278-401 |
| 平均耗时 | 68.5s | 61.7s |

**关键洞察**：
- 中文版共识率更高（可能是中文 SOUL prompt 更明确）
- 中文版 Token 数更少（输出更简洁）
- 蝙蝠球题在中文场景下两个 SOUL 都对了，说明 SOUL steering 在中文场景同样有效

## 🔧 技术瓶颈

### 瓶颈 1: 答案提取准确率

**问题**: 4/16 题目答案提取失败或错误

**案例**：
- Q1（17只羊）中文版：忒弥斯仲裁说"歧义"，实际答案是 9
- Q6（3加仑5加仑）中文版：仲裁文本完整但未提取出 "4"
- Q7（电商ROAS）中文版：应该有 ROAS 数字但未提取

**根因**: 正则表达式未覆盖中文表达习惯

**改进方向**:
- [ ] 增加中文显式标记检测（"答案是"、"结论是"、"因此"）
- [ ] 提取最后一个数字 + 单位（4加仑、5分钟）
- [ ] LLM 辅助提取（用小模型单独提取答案）

### 瓶颈 2: Stage 1 真并行 vs 伪并行

**当前实现**: 串行两次 `generate()`（GIL 释放但仍是串行）

**实测**: Code ~5-13s，Crit ~28s，总计 ~33-41s

**改进方向**: MLX Batch Forward (B=2)
- ADR-132 已验证 B=2 加速 1.97x
- 预期：41s → 21s（减少 20s）

### 瓶颈 3: 共识率偏低（37.5%）

**原因**:
- 12B 模型单次推理不够稳定
- Code 和 Crit 的 prompt 差异较大（导致答案不一致）

**改进方向**:
- [ ] 升级到更大模型（如 Llama-3-70B）
- [ ] 调整 SOUL prompt 使其更接近（只保留关键差异）
- [ ] 增加 Stage 1 的 SOUL 数量（3个投票，少数服从多数）

## 💡 成功案例

### 案例 1: 蝙蝠球题（中文版）⭐⭐⭐

**题目**: 蝙蝠和球一共 1.10 元。蝙蝠比球贵 1.00 元。球多少钱？

**Stage 1 并行推理**（38.0s，共识达成）：
- Code: 145 tok，10.6s → **$0.05** ✅
- Crit: 392 tok，27.4s → **0.05元** ✅

**共识**: ✅ 直接输出，无需仲裁

**分析**:
- 两个 SOUL 都正确理解了线性方程组（x + (x+1) = 1.10）
- Crit 在中文场景下的推理能力优于英文场景

### 案例 2: 100台机器题（英文版）

**题目**: 如果 5 台机器 5 分钟可以做 5 个零件，100 台机器做 100 个零件需要多久？

**Stage 1 并行推理**（35.1s）：
- Code: 92 tok，7.1s → **5 minutes** ✅
- Crit: 401 tok，28.0s → 100 ⚠️（理解错误）

**Stage 2 仲裁**（25.3s）：
- 忒弥斯: 325 tok → **5 minutes** ✅

**总耗时**: 60.3s，最终答案正确

**分析**:
- Code 正确识别了并行关系（5台机器5分钟 = 每台机器1分钟1个零件）
- Crit 误以为需要更多时间
- 忒弥斯仲裁修正了 Crit 的错误

## 🎯 结论

### ✅ 验证成功的假设

1. **双阶段混合策略有效**
   - Stage 1 共识 = 高置信度（6/6 正确）
   - Stage 1 分歧 → Stage 2 仲裁（大部分修正为正确）

2. **SOUL Steering 在 12B 模型上有效**
   - Code: 数学推理准确（90%+）
   - Crit: 假设验证能力强
   - 忒弥斯: 仲裁效果好（70%+）

3. **中英文场景都可用**
   - 中文版共识率更高（50% vs 25%）
   - 蝙蝠球题在中文场景下 Code + Crit 都对了

### ⚠️ 待解决的问题

1. **答案提取准确率**（75% → 目标 95%）
2. **Stage 1 真并行**（串行 41s → 并行 21s）
3. **共识率提升**（37.5% → 目标 60%+）

### 🚀 下一步

**P1 优先级**:
1. 修复答案提取（中文正则优化）
2. 实施 MLX Batch Forward (B=2) 真并行
3. 增加测试题库（20+ 题）验证稳定性

**P2 长期**:
- 升级到 Llama-3-70B（提升共识率）
- 3-SOUL 投票机制（少数服从多数）
- 探索其他 SOUL 组合（雅典娜 + Code 分析类？）
